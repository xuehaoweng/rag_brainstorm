# 20 分钟入门 RAG：让大模型别急着回答，先去查资料

你有没有遇到过这种情况：

你问 AI 一个项目问题，它回答得很自信，格式也很漂亮。

但仔细一看，路径是错的，命令是编的，引用的文档根本不存在。

这不是因为大模型“不聪明”，而是因为它缺少一件事：**证据**。

RAG 就是为了解决这个问题而出现的。

RAG，全称 `Retrieval-Augmented Generation`，中文通常叫“检索增强生成”。不用被名字吓到，它的核心只有一句话：

**先从你的知识库里查资料，再让大模型基于资料回答。**

也就是说，RAG 不是让模型“更会猜”，而是让模型“有依据地说”。

这篇文章用 20 分钟带你理解三件事：

- RAG 为什么重要。
- 一个 RAG 系统到底怎么工作。
- 它能在个人知识库、研发、客服和企业内部问答里提高什么效率。

为了不只讲概念，我会结合当前项目 `Self RAG` 来说明。这个项目是一个个人 Markdown RAG 学习控制台，它的重点不是把答案包装得多漂亮，而是把 RAG 的关键过程摊开给你看：文档扫描、文本切分、向量化、关键词检索、混合检索、证据来源和调试结果。

换句话说，它更像一个“RAG 证据实验室”。

## 1. 为什么需要 RAG

大模型有三个天然限制。

第一，它不知道你的私有资料。

公司的 SOP、项目 README、个人笔记、会议纪要、接口文档、故障记录，这些内容通常不在模型训练数据里。你不把资料给它，它就只能靠已有知识和概率推断。

第二，它的知识可能过期。

模型训练完成之后，新的业务变更、代码改动、产品策略、线上事故、内部规范，不会自动进入它的大脑。

第三，它可能编造答案。

尤其是当问题涉及文件路径、版本号、配置项、命令、日期、接口参数时，如果没有外部证据，模型很容易生成一个“看起来合理，但实际错误”的回答。

RAG 的思路很朴素：

**不要让模型凭记忆回答。让它先查资料。**

查到证据后，再让模型做总结、归纳和表达。

所以，RAG 不是替代大模型，而是给大模型接上一个“可检索的外部记忆”。

## 2. RAG 的基本工作流

一个最小可用的 RAG 系统，通常有两条链路。

第一条是索引链路，负责把资料整理成机器可以检索的结构：

```text
原始文档 -> 文档加载 -> 文本切分 -> Embedding 向量化 -> 存入索引
```

第二条是问答链路，负责根据用户问题找到证据，再生成答案：

```text
用户问题 -> 检索相关文本块 -> 构造上下文 -> 大模型生成答案 -> 返回引用来源
```

当前项目 `Self RAG` 走的也是这条路线：

```text
Markdown Folder
  -> Document Loader
  -> Markdown Chunker
  -> Vector Index + Keyword Index
  -> Retriever
  -> Debug Web UI
```

你可以把它理解成一个可视化实验台：

输入一个 Markdown 文件夹和一个问题，系统会告诉你它扫描了哪些文件、切出了哪些文本块、检索到了哪些证据、每条证据的分数是多少。

这对初学者很重要。因为很多人第一次学 RAG，只看到最后那段“AI 生成的答案”，却看不到答案背后的证据链。

但真正决定 RAG 质量的，往往不是最后生成那一步，而是前面有没有找对资料。

## 3. 第一步：先把文档装进来

RAG 的第一步不是调用大模型，而是整理资料。

在当前项目里，资料来源是一个本地 Markdown 文件夹。后端会递归扫描 `.md` 文件，读取文件内容，并记录文件路径、内容哈希、大小和修改时间。

这一步看起来简单，但它决定了一个非常关键的问题：

**你的知识库边界在哪里？**

RAG 只能基于你给它的资料回答。资料里没有的内容，它就不应该假装知道。

一个好的 RAG 系统，应该敢于说：

```text
当前证据不足，无法回答。
```

这句话听起来不够“智能”，但它比编一个错误答案有价值得多。

## 4. 第二步：把长文档切成合适的 chunk

为什么不能直接把一整篇文档塞给大模型？

因为文档可能很长，模型上下文窗口有限，而且检索需要足够细的粒度。

切得太大，检索结果里会混入大量无关内容。

切得太碎，语义又会断裂，模型拿到的证据不完整。

这就是文本切分，也就是常说的 `chunking`。

当前项目的 `Markdown Chunker` 采用结构感知切分：

- 优先按 Markdown 标题切分。
- 短章节尽量保持完整。
- 长章节按段落继续拆分。
- 避免切开代码块。
- 保留文件路径、标题路径、行号范围、文本哈希等元数据。

默认参数里，单个 chunk 最大约 `1800` 个字符，重叠约 `160` 个字符。

这里的“重叠”是为了解决边界问题。比如一个关键结论刚好出现在两个文本块交界处，如果完全不重叠，检索时可能只召回半句话；保留少量重叠，可以提高召回完整上下文的概率。

很多 RAG 效果不好，并不是模型不行，而是文档切分不合理。

这也是为什么 `Self RAG` 会把 chunk 预览做出来：你必须先看见系统怎么切文档，才知道后面的检索为什么好或不好。

## 5. 第三步：用 Embedding 表示语义

Embedding 是 RAG 里最重要的概念之一。

它会把一段文本转换成一组数字，也就是向量。

直观理解就是：语义相近的文本，向量距离更近；语义不相近的文本，向量距离更远。

例如用户问：

```text
如何减少模型幻觉？
```

文档里可能没有“幻觉”这个词，而是写着：

```text
回答必须基于证据，并引用来源；证据不足时不要编造。
```

关键词检索可能匹配不到，但向量检索有机会判断它们语义相关。

当前项目支持两类 Embedding Provider：

- `hash`：用于本地测试和无依赖演示，不是真正的语义模型。
- `bge-m3` / `sentence-transformers`：用于真实语义检索，需要本地模型。

这也很适合学习。

你可以先用 `hash` 跑通完整链路，确认系统能扫描、切分、索引和检索。等流程理解清楚后，再换成真实 Embedding 模型，观察语义检索带来的变化。

## 6. 第四步：检索，RAG 的核心战场

RAG 的核心不是“生成”，而是“找对证据”。

当前项目支持三种检索模式：关键词检索、向量检索和混合检索。

### 关键词检索：适合找精确内容

关键词检索适合查这些内容：

- 文件名
- 函数名
- 命令
- 错误码
- 日期
- 专有名词
- 版本号

比如你要查 `ERR_CONNECTION_RESET`、`pgvector`、`2026-04-23`，关键词检索通常很可靠。

它的缺点也很明显：需要字面匹配。

如果用户换了一种说法，文档里又没有相同关键词，就可能搜不到。

### 向量检索：适合找语义相关内容

向量检索适合查概念性问题。

它不要求用户问题和文档使用完全相同的词。只要语义接近，就有机会召回。

比如用户问“怎么降低 AI 胡说八道的概率”，文档里写的是“证据不足时不要编造”，向量检索就可能把它们关联起来。

它的缺点是对精确标识符不一定敏感。函数名、路径、版本号、错误码这类内容，在向量模型眼里可能只是普通字符串。

### 混合检索：更适合真实知识库

混合检索就是把关键词检索和向量检索结合起来。

当前项目里的 `HybridRetriever` 会分别执行向量检索和关键词检索，然后归一化两边的分数，再按权重合并结果。项目当前默认更偏向关键词分数：

```text
final_score = 0.35 * vector_score + 0.65 * keyword_score
```

这个权重很适合个人知识库和项目文档。

因为真实文档里经常同时存在概念、文件名、命令、日期和代码片段。纯向量检索容易漏掉精确信息，纯关键词检索又容易漏掉语义相关内容。

实际使用时，混合检索通常更稳。

## 7. 第五步：把证据交给大模型

检索到文本块后，系统会把这些文本块组织成上下文，交给大模型。

大模型看到的不再只是用户问题，而是类似这样的输入：

```text
用户问题：RAG 混合检索有什么用？

参考资料：
[chunk 1] 来自 docs/xxx.md，标题路径：Retrieval > Hybrid
...

[chunk 2] 来自 app/retrieval/hybrid.py
...

请只基于参考资料回答，并标注引用来源。
```

这样做有三个好处。

第一，答案更贴近你的资料，而不是泛泛而谈。

第二，回答可以带引用，方便你回到原文核对。

第三，当资料不足时，系统可以暴露证据缺口，而不是强行编造。

当前项目还处在学习控制台阶段，所以它重点展示检索结果、分数、来源和 chunk 文本。

这不是“半成品体验”，而是学习 RAG 最有价值的部分。

如果你能看懂证据是怎么被召回的，就能判断一个 RAG 系统为什么答得好，也能判断它为什么答错。

## 8. RAG 适合哪些场景

RAG 特别适合一类问题：

**资料很多、问题重复、答案需要基于事实。**

### 个人知识库问答

如果你有大量 Markdown 笔记、读书笔记、项目记录、学习计划，RAG 可以帮你快速从个人资料里找答案。

比如：

- “我之前怎么规划 RAG 学习路线的？”
- “这个项目的混合检索策略是什么？”
- “哪些文档提到过 bge-m3？”

### 企业内部知识助手

公司里的知识经常散落在文档、Wiki、飞书、Confluence、代码仓库和工单系统里。

RAG 可以把这些资料接入一个统一的问答入口。

常见问题包括：

- “这个接口怎么鉴权？”
- “上线流程有哪些步骤？”
- “某个故障以前怎么处理的？”

### 客服和售后支持

客服场景有大量重复问题，例如退款规则、套餐差异、安装步骤、故障排查。

RAG 可以基于 FAQ、产品文档和历史工单生成回答。

它的价值不是完全替代人工，而是先给出带依据的草稿，让人工审核、修改和发送。

### 研发效率工具

对开发团队来说，RAG 可以接入 README、设计文档、API 文档、代码注释、故障复盘和测试说明。

它可以回答：

- “这个模块的入口在哪里？”
- “如何本地启动项目？”
- “某个错误日志可能对应哪个组件？”
- “这个设计为什么选择混合检索？”

### 数据分析和运营复盘

如果把日报、周报、实验记录、用户反馈和指标解释文档接入 RAG，就可以快速查询历史结论。

例如：

- “上次转化率下降的原因是什么？”
- “哪些实验影响过注册流程？”
- “某个功能上线后的用户反馈有哪些？”

## 9. RAG 到底提高了什么效率

RAG 提高的不是某一个按钮的效率，而是这件事的效率：

**从一个问题，到一个可信答案。**

### 更快找到资料

传统方式是你自己打开文件夹、搜索关键词、翻文档、判断哪段有用。

RAG 可以先帮你召回候选证据，再由模型总结。

资料越多，这一步越有价值。

### 让旧知识重新可用

很多组织和个人的问题不是没有答案，而是答案埋在旧文档里。

RAG 可以让过去写过的材料重新进入工作流。

一次沉淀，多次复用。

### 降低新人上手成本

新人不熟悉项目结构和历史背景，常常不知道该问谁、去哪找。

RAG 可以作为项目问答入口，让新人通过自然语言了解系统。

比如当前项目就可以问：

```text
这个项目如何进行 Markdown 切分？
关键词检索和向量检索有什么区别？
前端如何调用后端检索接口？
```

### 减少重复支持工作

客服和支持人员不需要每次从头组织答案。

RAG 可以根据知识库生成初稿，并附带来源。人工只需要审核、修改和发送。

这能减少重复劳动，也能降低回答不一致的风险。

### 提高排错和决策质量

当答案带有来源时，你可以快速追溯依据。

对技术排错、方案评审、业务复盘来说，这比只给一个结论更有价值。

RAG 的理想输出不是“听起来很聪明”，而是：

**可核对、可追溯、可复用。**

## 10. 用当前项目跑一次 RAG 实验

如果你想用当前项目体验 RAG，可以先启动后端和前端：

```bash
bash scripts/dev-backend.sh
bash scripts/dev-frontend.sh
```

然后打开：

```text
http://<server-ip>:5173/
```

在页面上可以配置：

- Markdown 根目录
- 检索问题
- 检索模式：`keyword`、`vector`、`hybrid`
- Embedding Provider：`hash` 或 `bge-m3`
- Top K 返回条数
- Chunk 最大字符数

建议按这个顺序学习：

1. 先点“预览相关切分”，看 Markdown 被切成了哪些 chunk。
2. 再用 `keyword` 搜一个精确词，比如文件名或函数名。
3. 再用 `vector` 搜一个概念性问题，观察语义召回效果。
4. 最后用 `hybrid` 对比两边结果，看混合检索是否更稳。

如果检索效果不好，不要第一时间怀疑大模型。

先检查三个问题：

- 文档里是否真的有答案？
- chunk 是否切得太大或太碎？
- 查询方式更适合关键词、向量还是混合检索？

这个检查顺序很实用。因为 RAG 问题通常不是“模型一句话没写好”，而是证据链前面某一环出了问题。

## 11. 初学者最容易误解的 4 件事

### 误解一：RAG 等于向量数据库

不是。

向量数据库只是 RAG 的一个组件。完整 RAG 还包括文档处理、切分、Embedding、检索、重排、上下文构造、答案生成、引用和评估。

### 误解二：用了 RAG 就不会幻觉

也不是。

RAG 可以降低幻觉，但不能自动消灭幻觉。检索错了、上下文拼错了、提示词约束不够，模型仍然可能答错。

### 误解三：chunk 越小越好

不是。

chunk 太小会丢上下文，chunk 太大又会降低检索精度。合适的切分策略要结合文档结构和问题类型调整。

### 误解四：向量检索一定比关键词检索高级

不一定。

查命令、路径、错误码、版本号时，关键词检索往往更可靠。实际项目里，混合检索通常比单一检索更实用。

## 12. 总结：RAG 的关键不是回答，而是证据

RAG 的本质，是让大模型从“凭记忆回答”变成“基于证据回答”。

它的基本流程是：

```text
整理资料 -> 切分文本 -> 生成向量 -> 检索证据 -> 构造上下文 -> 生成带来源的答案
```

它适合个人知识库、企业内部问答、客服支持、研发排错、运营复盘等场景。

它能提升的核心效率，是从大量资料中快速找到可信答案的效率。更进一步，它还能提高知识复用效率、新人上手效率、客服响应效率和技术排错效率。

如果你想真正理解 RAG，不要只看最终答案。

更应该像当前 `Self RAG` 项目这样，把 chunk、检索分数、来源路径和原文证据都展示出来。

因为 RAG 系统的关键问题永远是：

**回答之前，证据找对了吗？**
