From zero to one

用当前项目学懂 RAG。

这个项目不是把 RAG 封装成黑盒，而是把“资料怎么进来、怎么切、怎么变成向量、怎么检索、怎么带引用回答”摊开给你看。

看源码对应关系 按路线学习
1. Markdown 知识库 本地资料是系统能回答的边界。
2. Chunk 切分 把长文档变成可检索的证据块。
3. Embedding + 索引 把文本转成向量，存入 SQLite + FAISS。
4. Hybrid Retrieval 关键词抓精确项，向量抓语义项。
5. Grounded Answer 只基于证据回答，并标注来源。

当前项目能学到什么

它覆盖了一个最小但完整的本地 RAG 系统。你能看到每一步的输入、输出和失败点，所以很适合从 0 到 1 理解 RAG。

Ingestion

文档加载

递归扫描 Markdown，记录路径、哈希、修改时间和大小。你会理解知识库边界为什么重要。

Chunking

结构化切分

按标题、段落和代码块切分，并保留标题路径与行号。你会看到 chunk 太大或太碎的影响。

Embedding

语义向量

支持 hash 演示和 bge-m3 真实语义模型。你会理解“语义相近”如何变成向量相近。

Index

持久化索引

SQLite 保存文档和 chunk 元数据，FAISS 保存向量索引。你会理解为什么不能每次提问都重新处理全量资料。

Retrieval

关键词、向量、混合检索

关键词适合精确字符串，向量适合语义问题，混合检索适合真实知识库。

Generation

带引用生成

先把检索证据组装进 prompt，再要求模型只基于证据回答。资料不足时应该拒答。

RAG 到底是什么

RAG 的全称是 Retrieval-Augmented Generation，中文常叫检索增强生成。它不是让模型更会猜，而是让模型先查资料。

RAG = 先从知识库检索证据，再让大模型基于证据生成答案。 一句话判断：如果没有检索到可靠证据，系统就不应该装作知道。
01Markdown项目文档、笔记、规范、故障记录。
02Loader扫描文件并生成文档对象。
03Chunker切成带行号的证据块。
04Embedding把文本转成数字向量。
05Index保存元数据和向量索引。
06Retriever根据问题召回相关 chunk。
07LLM基于证据回答并附来源。

三种检索怎么选

初学 RAG 时，最容易把问题归因给大模型。实际上，大量失败发生在检索阶段：没有找对证据，后面再强的模型也会答偏。

Keyword

关键词检索

适合文件名、函数名、命令、版本号、错误码和精确中文短语。缺点是用户换一种说法时容易漏。

Vector

向量检索

适合概念性问题和同义表达。缺点是对路径、代码符号、版本号这类精确标识符不一定敏感。

Hybrid

混合检索

当前项目默认把关键词和向量结果归一化后融合。真实知识库里通常更稳，因为问题往往同时包含概念和精确词。

源码导览

点击左侧文件，右侧会显示它在 RAG 链路里的角色。你可以按这个顺序读源码，不需要一上来就钻进所有细节。

关键词速查

先把这些词弄清楚，读 RAG 代码会轻松很多。每个词都尽量对应到当前项目里的具体行为。

RAG
检索增强生成。先从知识库找证据，再把证据交给大模型回答。当前项目的 `/api/answer` 就是完整 RAG 问答入口。
Grounding
让答案“落在证据上”。不是让模型自由发挥，而是要求它只基于检索到的 chunk 回答，并在事实句后加来源。
Document
一篇原始资料。当前项目里通常是一份 Markdown 文件，比如 `notes/articles/*.md`。
Chunk
从长文档切出来的小文本块。检索时召回的是 chunk，不是整篇文档。
Metadata
描述 chunk 的附加信息，例如文件路径、标题路径、起止行号和文本哈希。
Embedding
把文本转换成一组数字向量。语义越接近，向量距离通常越近。
Vector
一串数字，表示文本的语义位置。项目里向量会被归一化后用于相似度搜索。
Vector Store
向量索引库。当前项目用 FAISS 存向量，用 SQLite 存 chunk 原文和元数据。
FAISS
高效向量搜索库。项目里用它根据 query vector 找最相似的 chunk vector。
SQLite
本地数据库。项目里保存 documents、chunks 和 index_metadata。
Query
用户问题。检索时 query 也会被转换成向量，和 chunk 向量进行比较。
Retriever
检索器。负责从所有 chunk 中找出与 query 最相关的一批候选证据。
Keyword Search
关键词检索。适合找路径、命令、函数名、配置项、错误码等精确字符串。
Vector Search
向量检索。适合找语义相关内容，即使问题和文档用词不完全一样。
Hybrid Search
混合检索。把关键词结果和向量结果合并，项目默认关键词权重更高。
Top K
最多返回多少条检索结果。Top K 越大，证据更全，但噪声也可能更多。
Score
检索相关性分数。它不是概率，只能用来比较同一次检索里的相对排序。
Recall
召回。意思是“应该找到的证据有没有被找出来”。RAG 答错常常是召回失败。
Precision
精度。意思是“找出来的结果里有多少是真的有用”。Top K 过大时精度常会下降。
Reranker
重排器。先粗检索一批候选，再用更精细的模型重新排序。当前路线图里计划后续加入。
Prompt
发给大模型的指令和上下文。项目里 prompt 会包含来源编号、chunk 原文和用户问题。
Context
模型回答时能看到的参考材料。RAG 的 context 应该来自检索结果，而不是凭空补充。
Citation
引用来源。当前项目用文件路径、标题路径和行号说明答案依据来自哪里。
Hallucination
幻觉。模型编出看似合理但没有证据支持的内容。RAG 用证据和引用降低这个风险。
No Answer
拒答或证据不足。当知识库没有可靠证据时，系统应该返回“当前知识库证据不足。”
Provider
具体服务实现。比如 embedding provider 可以是 `hash`，也可以是 `bge-m3`。

从 0 到 1 学习路线

不建议一开始就追求“问答效果很酷”。先把证据链跑通，再调 chunk、embedding、检索权重和引用生成。

理解文档和 chunk

先跑 `/api/index/preview`，观察同一篇 Markdown 被切成了哪些 chunk，标题路径和行号是否保留完整。

比较关键词和向量检索

用文件名、命令、概念问题分别测试 keyword、vector、hybrid，观察哪个模式召回更准。

理解 embedding 的作用

先用 hash 跑通流程，再换 bge-m3，比较同义表达和精确技术词的召回差异。

构建持久化索引

调用 `/api/index/build`，理解为什么 SQLite 存元数据、FAISS 存向量，以及 provider 不一致时为什么要重建索引。

学习 grounded generation

看 `app/generation/prompt.py` 如何把 chunk 格式化成 `[来源 N]`，以及为什么每个事实句都要有引用。

推荐练习问题

用这些问题跑一遍前端控制台，可以快速建立 RAG 调试直觉。

  • chunk 为什么要保留标题路径和行号？
  • 为什么 hash embedding 只能用于演示，不能代表真实语义检索？
  • 搜索 `SELF_RAG_LLM_API_KEY` 时，关键词检索和向量检索谁更可靠？
  • 问“如何减少模型幻觉”时，hybrid 是否能召回 prompt 约束？
  • 如果证据不足，系统应该返回什么？
  • 为什么持久化索引的 provider 和查询 provider 必须一致？
